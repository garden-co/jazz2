Permissions control who can read, insert, update, and delete rows. Jazz enforces them server-side using row-level policies defined in `permissions.ts`. Clients that fail a policy check have their writes rejected and their reads filtered.

Locally, Jazz distinguishes between two runtime states:

- no compiled policy bundle loaded: local session-scoped reads and writes stay permissive
- compiled policy bundle loaded: every read, insert, update, and delete needs an explicit grant, or it is denied

Apps that do not need user-scoped filtering should still declare explicit grants (`policy.todos.allowRead.always()` or `policy.todos.allowRead.where({})`) once a compiled bundle is loaded.

### Inherited access (`allowedTo.*`)

A row can inherit its access from a related row. Use
`allowedTo.read/insert/update/delete(...)` to express that inheritance.

### Share-based access

Sometimes access isn't determined by ownership or a parent relationship, but by a separate "shares" table. Use `policy.<table>.exists.where(...)` to check whether a matching row exists in another table.

</include>

The callback form `(todo) => ...` gives you access to the current row, so you can correlate it with rows in other tables.

### Update policies: old row vs new row

Update policies can check both the row **before** the update (`.whereOld(...)`) and the row
**after** the update (`.whereNew(...)`). This is useful when you need to verify that the user had
permission to modify the original row and that the result is also valid.

<include cwd lang="ts" meta='title="permissions.ts"'>
  ../examples/docs/todo-server-ts/src/permissions-snippets.ts#permissions-whereold-wherenew-ts
</include>

If you only use `.whereOld(...)`, the same condition is applied to both the old and new row. The
same applies if you only use `.whereNew(...)`. Use both when the old-row and new-row checks differ.

## Policy enforcement and request context

Every policy is evaluated against a session. On the frontend, this is the authenticated user's session. In a backend handler, create a scoped session per request so queries run with the right identity.

<include cwd lang="ts" meta='title="handler.ts"'>
  ../examples/docs/todo-server-ts/src/request-context.ts#backend-request-handler-ts
</include>

See [Server Setup](/docs/getting-started/server-setup#backend-context-setup) for backend context setup details.

<Callout type="info">
  Structural-only client runtimes stay permissive locally so offline reads and writes keep working.
  Once a compiled bundle is loaded, Jazz enforces explicit grants locally too. Sync servers still
  reject violating writes, and server-scoped reads are filtered before data is sent to the client.
</Callout>

## Checking write permissions before showing UI

`insert`, `update`, and `delete` remain the authority. You can call them directly and let Jazz reject
writes that do not pass policy. For controls such as disabled buttons, hidden menu items, and
read-only notices, Jazz also exposes preflight helpers that ask the local runtime whether an insert
or update would currently be allowed.

These helpers are consultative. They do not reserve the permission result, they do not wait for a
remote authority, and they can return `"unknown"` when the local runtime does not yet have enough
state to decide. Always keep the write path prepared for rejection.

### Direct `Db` API

Use `db.canInsert(table, data)` with the same table proxy and insert input shape you pass to
`db.insert(...)`. Use `db.canUpdate(table, id, patch)` with the same table proxy, row id, and patch
shape you pass to `db.update(...)`.

Both methods return `Promise<true | false | "unknown">`.

For example, this policy only lets admins create announcement rows:

```ts title="permissions.ts"
export default definePermissions(app, ({ policy, session }) => {
  const isAdmin = session.where({ "claims.role": "admin" });

  policy.announcements.allowRead.always();
  policy.announcements.allowInsert.where(isAdmin);
  policy.announcements.allowUpdate.where(isAdmin);
  policy.announcements.allowDelete.where(isAdmin);
});
```

To decide whether to show the "New announcement" button, pass a draft row with the same shape you
would eventually insert:

```ts
const announcementDraft = {
  title: "",
  body: "",
  published_at: new Date(),
};

const canCreateAnnouncement = await db.canInsert(app.announcements, announcementDraft);
const showNewAnnouncementButton = canCreateAnnouncement === true;

// The write still performs the authoritative permission check.
await db.insert(app.announcements, realAnnouncement).wait({ tier: "edge" });
```

This draft-data pattern is a workaround for insert rules: insert policies evaluate the row being
created, so Jazz needs candidate values even when you only want to decide whether a button should be
visible. Use harmless placeholder values for fields that do not affect the policy, and use realistic
values for fields that do.

`false` means the current local permission state denies the write. `"unknown"` means Jazz cannot
prove the write is allowed or denied from local state alone, for example because the target row,
schema, or permission context is not loaded yet.

<Callout type="warn">
  Do not use `canInsert` or `canUpdate` as your only protection. They are useful for UI affordances,
  but a policy can change, local state can be incomplete, and the write itself is still the
  operation that must be accepted or rejected.
</Callout>

### Framework helpers

Framework bindings wrap the direct `Db` calls and expose a reactive loading state. The helper returns
`"unknown"` until the underlying promise resolves, then returns the same `true | false | "unknown"`
decision as `db.canInsert` or `db.canUpdate`.

    body: "",

## Magic columns

Jazz exposes a small set of system-provided magic columns at query time. They do not exist in your
schema, and they are omitted from `select("*")`, so opt in explicitly when you want them.

### Permission introspection columns

- `$canRead`&hairsp;—&hairsp;whether the current session can read the row. Since read policies already gate which rows appear, this will usually be `true`.
- `$canEdit`&hairsp;—&hairsp;whether the current session passes the row's update policy.
- `$canDelete`&hairsp;—&hairsp;whether the current session passes the row's delete policy.
- Without a session, all three return `null`.

### Edit metadata columns

Jazz also tracks row authorship and timestamps automatically:

- `$createdBy`&hairsp;—&hairsp;the Jazz principal that created the row
- `$createdAt`&hairsp;—&hairsp;when the row was first created
- `$updatedBy`&hairsp;—&hairsp;the Jazz principal that last updated the row
- `$updatedAt`&hairsp;—&hairsp;when the row was last updated

Jazz always tracks this metadata; select the columns explicitly to include them in query results. Backend writes that are not attributed to a user are authored as `jazz:system`.

### Authorship-based policies

You can use these edit metadata magic columns directly in `permissions.ts`. This is useful for simple
"creator can read/edit/delete their own rows" policies without adding explicit `owner_id` columns.

For the most common case, Jazz also exposes `policy.<table>.managedByCreator()` and `isCreator` as
shorthand for the same `$createdBy === session.user_id` condition. You can still write the raw
`$createdBy` comparison yourself whenever you want the explicit form.
